# ADR-001: Stored Procedures as the Exclusive Data Access Strategy

**Date:** Discovered 2026-07-20 (implicit decision, original date unknown)  
**Status:** Accepted (implicit — no record of formal decision)  
**Deciders:** Original development team (PSH, attributed in DataAccess class comments, ~2003)

---

## Context

The BSE System needs to read and write data to a SQL Server database. A decision was made about how application code would interact with the database.

Evidence in code:
- `legacy/DataAccessLib/clsDataAccess.vb` — all public methods accept a stored procedure name or SQL text but the only pattern observed across the entire codebase is named stored procedures.
- `legacy/BSEDatabase/dbo/Stored Procedures/` — approximately 230 stored procedures exist for every data operation (Add*, Edit*, Delete*, Get*, Copy*, Change*, Move*).
- No ORM (Entity Framework, Dapper, LINQ to SQL) is present in any project reference.
- No ad-hoc SQL strings are constructed in application code.
- The `BSEDatabase` SQL Server database project tracks every stored procedure under source control.

## Decision

All database interactions are performed by calling named stored procedures. No ORM, no ad-hoc SQL, no direct table access from application code.

## Consequences

**Observed positive effects:**
- All database logic is auditable in a single location (the SQL project).
- The stored procedure surface can be locked down with database-level permissions.
- Business-rule constraints (20+ CHECK constraints on the `Case` table) are enforced at the database tier independently of the application.

**Observed negative effects:**
- Schema changes require coordinated deployment of both the stored procedure and the calling application code.
- DataSet table ordering (e.g., `GetCaseDetailsByRBSE` returning 11 tables in a fixed order) is not type-safe; the mapping is expressed only as integer constants in `clsCase.vb`.
- Error handling inside stored procedures uses the pre-SQL-Server-2005 `@@ERROR` pattern rather than `TRY/CATCH/THROW`, making partial failure recovery unpredictable (observed in `AddCase.sql`, `MoveCase.sql`, `ChangeRBSE.sql`).
- The approximately 230 stored procedures represent a large volume of database-side logic that is not unit-testable without a live database connection.
