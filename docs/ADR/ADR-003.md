# ADR-003: Windows Integrated Authentication with Role-per-Page Authorisation

**Date:** Discovered 2026-07-20 (implicit decision, original date unknown)  
**Status:** Superseded by [ADR-006](ADR-006.md) ŌĆö Windows Auth is replaced with OIDC in the modernised system  
**Deciders:** Original development team

---

## Context

The BSE System is an internal government application deployed within the APHA/Defra network. Users are government employees with Windows domain accounts. Access needs to be restricted by user role (data entry vs. maintenance vs. read-only).

Evidence in code:
- `legacy/BSESystem/Web.config` sets `<authentication mode="Windows"/>` and `<deny users="?"/>`.
- `legacy/BSELib/clsUser.vb` ŌĆö `GetGroupForUser()` calls the `GetGroupForUser` stored procedure, passing the Windows NT login name and receiving a `UserGroup` integer and group name string.
- `legacy/BSESystem/VLAHeader.ascx.vb` ŌĆö `GetUserDetails()` runs on every page load to populate `Session[SV_HeaderUserID]`, `Session[SV_HeaderGroupName]`, `Session[SV_HeaderUserName]`.
- `legacy/BSESystem/ADNSExportGB.aspx.vb` ŌĆö `EnableControls()` redirects users based on string comparison against known group name literals: `"DEFRA Viewer"`, `"DEFRA Data Entry"`, `"DEFRA Maintenance"`, `"VLA Data Entry"`, `"VLA Maintenance"`.
- The `User` table (`legacy/BSEDatabase/dbo/Tables/User.sql`) stores `NTLogin VARCHAR(25)` as the identifier, linked to `luUserGroup`.

## Decision

Authentication is delegated entirely to Windows/IIS (NTLM/Kerberos). Authorisation is implemented by resolving the authenticated Windows identity to an internal user group (via the `User` and `luUserGroup` database tables) and then performing per-page string comparison of the group name against known literals.

## Consequences

**Observed positive effects:**
- No password management within the application.
- Consistent with the internal government network security model in use at deployment time.
- Group membership is stored in the database, allowing administrators to change user groups without redeploying the application.

**Observed negative effects:**
- **Fragile string comparison:** Authorisation uses case-sensitive string literals repeated in every restricted page (`"DEFRA Maintenance"`, etc.). A group name change in the database requires updating every affected ASPX page's `EnableControls()` method.
- **No centralised policy enforcement:** There is no `HttpModule`, `AuthorizeAttribute`, or single place that enforces access control. Each page independently decides whether to redirect. A newly added page with no `EnableControls()` call would be accessible to all authenticated users.
- **Session-based identity:** User identity is stored in session after first resolution. A session timeout followed by re-authentication would re-resolve the group, but mid-session group changes are not reflected until the session is cleared.
- **No claims or roles integration:** The system predates .NET claims-based identity and does not use `IPrincipal.IsInRole()`.
