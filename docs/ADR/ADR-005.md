# ADR-005: Data Access Strategy — Option A: Retain Stored Procedures with Dapper

**Date:** 2026-07-20  
**Status:** Accepted  
**Deciders:** Migration Programme — Human-approved decision  
**Supersedes:** N/A  
**Related:** ADR-001 (Stored Procedures as Exclusive Data Access Strategy — as-is discovery)

---

## Context

The BSE system's data access layer is built entirely on named stored procedures (see ADR-001). There are approximately **230 stored procedures** in `legacy/BSEDatabase/dbo/Stored Procedures/`, covering every data operation in the application (Add*, Edit*, Delete*, Get*, Copy*, Change*, Move*).

The modernisation programme requires a decision on how the new C# 14 / .NET 10 modular monolith should access the database.

**Option A — Retain Stored Procedures:** Keep all ~230 stored procedures as-is. The modernised application calls them via a Dapper thin wrapper over `Microsoft.Data.SqlClient`. No ORM is introduced. All SP source files are brought into version control under the new solution.

**Option B — Modernise to Entity Framework Core:** Replace stored procedure calls with EF Core DbContext, LINQ queries, and migrations. Stored procedures may coexist temporarily but are not the target state. Business logic currently in SPs would need to be understood and migrated to the application tier.

The migration programme has evaluated both options against the system characteristics:

| Factor | Option A impact | Option B impact |
|--------|----------------|----------------|
| Migration risk | Low — 230 SPs are a known quantity with proven business logic | High — SP business logic (20+ CHECK constraints, integer error codes, cascade operations) must be reverse-engineered and re-implemented |
| Business logic location | Remains split (SP + application) | Consolidated to application tier over time |
| Regression risk for regulatory outputs | Low — ADNS/OSS/BSESS paths call SPs directly; behaviour is unchanged | High — SP-to-LINQ conversion must produce identical regulatory outputs |
| Test effort | Lower — SPs are already behaviour-tested in production | Higher — full domain model must be built and verified against legacy SP behaviour |
| Timeline impact | Smaller — SP layer is not in scope for Phase 1–3 | Larger — EF migration would extend Phases 2–4 significantly |
| Database portability | Low — SPs are SQL Server-specific | Higher — EF Core supports multiple providers |
| Legacy SP debt | Carried forward | Eliminated over time |

---

## Decision

**Option A — Retain Stored Procedures with Dapper** is adopted for the modernised system.

All database interactions in the new C# 14 / .NET 10 codebase will be performed by calling existing named stored procedures via a Dapper thin wrapper. No ORM (Entity Framework Core, LINQ to SQL) is introduced. No ad-hoc SQL is permitted in application code.

---

## Implementation Detail

### Dapper Wrapper Pattern

The legacy `libDataAccess.DataAccess` and `libDataAccess.TBCultureDA` static classes are replaced by two injectable interfaces in `BSE.Infrastructure`:

```
IDbConnectionFactory
  CreateConnection() → SqlConnection

IDapperRepository
  QueryAsync<T>(spName, param?)          // replaces FillDataTable
  QuerySingleOrDefaultAsync<T>(...)      // single-row reads
  QueryMultipleAsync(spName, param?)     // replaces FillDataSet (multi-result SPs)
  ExecuteAsync(spName, param?)           // replaces ExecuteNonQuery
  ExecuteWithOutputAsync(spName, params) // replaces ExecuteQuery (OUTPUT params)
  ExecuteInTransactionAsync(work)        // explicit transaction scope
```

All methods use `CommandType.StoredProcedure`. No raw SQL strings are constructed in application code at any point.

### SP Source Control

All ~230 stored procedure `.sql` files are migrated from `legacy/BSEDatabase/dbo/Stored Procedures/` to `database/BSE.Database/dbo/Stored Procedures/` in **Slice 1** of the migration plan. This establishes a single source of truth for SP definitions.

**Version control rules:**
- No SP may be changed without a PR that includes the corresponding application code change
- PRs modifying SP result set ordering must include an impact assessment on all callers
- SP-to-domain mapping headers are added to every SP file in Slice 1

### Transaction Scope Fix (Critical)

A known defect in the legacy system (`UpdateCaseDetails` calling 8+ SPs across separate connections with no outer transaction — see ADR-004, Risk R01) is **corrected** in the migration. The `IDapperRepository.ExecuteInTransactionAsync()` method is used to wrap all case-save SP calls in a single `SqlTransaction`.

### Error Handling Refactor in SPs

Stored procedures using the legacy `@@ERROR` pattern (`AddCase`, `EditCase`, `ChangeRBSE`, `MoveCase`, `DeleteCase`) are refactored to `BEGIN TRY / BEGIN CATCH / ROLLBACK TRANSACTION` in Slice 8. This is the only SP code change made during the migration.

---

## Rationale for Option A

1. **Risk reduction:** The 230 stored procedures encode production-proven business logic including 20+ CHECK constraints on the `Case` table, complex cascade operations (`ChangeRBSE` across 15 tables, `ChangeCPHH` across 8 tables), and regulatory output generation. Rewriting this logic in LINQ would introduce significant regression risk for no behavioural benefit.

2. **Regulatory output safety:** The ADNS export, OSS export, and BSESS cross-check paths produce regulatory outputs with legal obligations. Calling the same stored procedures ensures byte-identical outputs without a complex output comparison programme.

3. **Timeline fit:** The migration programme is sized for 18 months. Adding EF Core migration to the scope would extend the programme by an estimated 6–12 months beyond the current plan.

4. **Team competency:** The team has existing knowledge of the stored procedure surface. A Dapper wrapper is a low-learning-curve change; EF Core requires significant upskilling on domain modelling, EF migrations, and LINQ query translation.

5. **Dapper suitability:** Dapper is a mature, production-proven micro-ORM with first-class `CommandType.StoredProcedure` support, OUTPUT parameter handling, and multi-result set reading (`QueryMultiple`). It replaces the bespoke `DataAccess`/`TBCultureDA` wrapper with a well-maintained open-source library.

---

## Consequences

### Positive

- Migration scope is well-bounded: application code is modernised; SP business logic is preserved unchanged.
- Regulatory output regression risk is minimised: the same stored procedures produce the same results.
- Dapper introduces testability that the legacy static `DataAccess` class lacked — `IDapperRepository` can be mocked in unit tests.
- Connection string is injected from environment variables, removing the deprecated `ConfigurationSettings.AppSettings` dependency.
- The `ExecuteInTransactionAsync` wrapper corrects the existing partial-update defect in `UpdateCaseDetails`.

### Negative and Trade-offs

- Business logic remains split between the application and the database tier. Stored procedures continue to be a source of "hidden" business rules that are not visible from the application code alone.
- The approximately 230 stored procedures are not unit-testable without a live database connection. Slice-level integration tests (not pure unit tests) are required to verify SP behaviour.
- The `@@ERROR` pattern in some SPs is only partially addressed (refactored in Slice 8 for the five most critical SPs). Remaining SPs continue to carry the pre-SQL-2005 error handling pattern.
- Future portability to a non-SQL-Server database remains constrained.

### Downstream Impact on Implementation Agent

- Every domain module must implement its data access via `IDapperRepository` — no static class calls, no direct `SqlConnection` construction.
- The `ExecuteInTransactionAsync` pattern is **mandatory** for `CaseManagement.UpdateCaseDetailsAsync()`. This is a go/no-go criterion for Slice 8.
- Named C# record types replace all `DataSet`/`DataTable`/integer-index access — this is a strict rule enforced in code review.

### Downstream Impact on UI Agent

- All data access in the new UI layer (Razor Pages / Blazor) routes through the injected module service interfaces, never directly through `IDapperRepository`.
- No `DataSet` or `DataTable` types appear in the UI layer.
- UI layer does not import `BSE.Infrastructure` directly.

---

## Review Date

This decision should be reviewed after Phase 4 (UI Migration and Cutover) is complete. At that point, an assessment of whether a selective EF Core migration for specific domains (e.g., UserManagement or ReferenceData) would provide net benefit can be made based on production operational experience.
