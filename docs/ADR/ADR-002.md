# ADR-002: ASP.NET InProc Session as the Multi-Step Data Transport

**Date:** Discovered 2026-07-20 (implicit decision, original date unknown)  
**Status:** Accepted (implicit — no record of formal decision)  
**Deciders:** Original development team

---

## Context

Case entry in the BSE System spans multiple web pages (Farm, Clinical, BAB, Feeds, Relations, VLA, DEFRA). The user navigates between pages in a defined sequence before committing all data to the database. A mechanism was needed to preserve partial data across this multi-page flow.

Evidence in code:
- `legacy/BSESystem/SessionVars.vb` defines 40+ session key constants.
- `legacy/BSESystem/CaseEntrySave.aspx.vb` reads `Session[SV_CaseDetails]` (a full 11-table `DataSet`) and `Session[SV_FarmDetails]` (a 3-table `DataSet`) immediately before calling `clsCase.UpdateCaseDetails()`.
- `legacy/BSESystem/Home.aspx.vb` calls `Common.GetCaseDetailsFromDatabase()`, which writes the complete `DataSet` to session on case load.
- The `clsADNSReport` object (a stateful class containing case lists and email body) is stored in `Session[SV_ADNSObject]` during the ADNS export flow (`ADNSExportGB.aspx.vb`).
- `Web.config` sets `<sessionState mode="InProc" ... timeout="20"/>` — session is held in the w3wp.exe worker process memory.

## Decision

The ASP.NET `HttpSessionState` is used as the primary state transport for all multi-page workflows. Complete `DataSet` objects (and in some cases full domain objects) are serialised into session between page navigations. The database is not written to until the user explicitly saves.

## Consequences

**Observed positive effects:**
- Simple to implement; no additional infrastructure required.
- Allows client-side navigation between pages without intermediate database writes.

**Observed negative effects:**
- **Single-server constraint:** InProc session is not shared across web servers. The application cannot be load-balanced without sticky sessions or session migration.
- **Data loss on recycle:** An IIS application pool recycle or server restart during a user's case-entry session discards all unsaved data with no recovery. The 20-minute timeout also silently discards data on idle sessions.
- **Memory pressure:** Multiple large `DataSet` objects (11-table case dataset, farm dataset, supplier datasets, ADNS object, batch datasets, etc.) per concurrent user session consume significant worker process memory.
- **No transactional integrity during navigation:** Each page stores modified DataSet rows back to session. If the browser is closed mid-flow, uncommitted rows are held in session until the 20-minute timeout expires — but the database is not affected.
- **Fragile session key coupling:** The 40+ session keys in `SessionVars.vb` are plain strings. Any typo in a key name at a call site silently returns `Nothing`, which can cause null-reference errors at runtime rather than compile time.
