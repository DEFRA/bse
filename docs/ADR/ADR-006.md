# ADR-006: Replace Windows Integrated Authentication with OIDC

**Date:** 2026-07-20  
**Status:** Accepted  
**Deciders:** Migration Programme — Human-approved decision  
**Supersedes:** ADR-003 (Windows Integrated Authentication with Role-per-Page Authorisation — as-is discovery; status updated to Superseded)  
**Related:** ADR-003, ADR-005, Target-Architecture.md §6

---

## Context

The legacy BSE system uses Windows Integrated Authentication (NTLM/Kerberos) enforced at the IIS level (ADR-003). This is hardcoded in `legacy/BSESystem/Web.config` (`<authentication mode="Windows"/>`, `<deny users="?"/>`) and is the only authentication mechanism in the system.

**Why Windows Auth cannot be retained in the modernised system:**

| Constraint | Detail |
|-----------|--------|
| Container deployment | Windows Integrated Authentication requires IIS (or IIS In-Process hosting on Windows). The target deployment is a Linux container running ASP.NET Core — Windows NTLM/Kerberos authentication is not available in this environment. |
| Framework replacement | ASP.NET Core's `Microsoft.AspNetCore.Authentication.Windows` is Windows-only and requires a Windows host process. Running on Linux (`.NET 10 / Kestrel`) precludes this option. |
| Load balancing | NTLM is connection-level; Kerberos requires a Windows domain account for the application service principal. Neither mechanism is viable in a horizontally-scaled container behind a reverse proxy without specialised GSSAPI configuration not supported in standard container deployments. |
| Modern standards | NTLM is considered deprecated by Microsoft. Government technical standards and the UK NCSC favour standards-based protocols (OAuth 2.0 / OIDC). |

**Current authorisation defects being corrected simultaneously (from ADR-003):**

1. Per-page string comparison (`"DEFRA Maintenance"`, `"VLA Maintenance"`, etc.) — fragile, unenforceable centrally.
2. Session-based identity (`SV_HeaderGroupName` in `HttpSessionState`) — group changes not reflected until session is cleared.
3. No `IPrincipal.IsInRole()` integration — predates .NET claims-based identity.

---

## Decision

**Windows Integrated Authentication is replaced with OpenID Connect (OIDC)** using `Microsoft.AspNetCore.Authentication.OpenIdConnect`.

The OIDC identity provider is **Azure Active Directory (Entra ID)** — the same Active Directory used by APHA/Defra, which already holds the same user accounts and domain memberships that the legacy system resolves via NT login names. This preserves the existing trust model (users authenticate with their government domain credentials) while enabling a standards-based protocol compatible with container deployment.

**Authorisation** is modernised simultaneously from per-page string comparison to centralised ASP.NET Core `AuthorizationPolicy` (registered in `BSE.Host/Program.cs`).

## Implementation Detail

### Authentication Flow

```
User (browser)
  │
  ▼
BSE.Host → redirect to Azure AD / Entra ID OIDC endpoint
  │         (user authenticates with government domain credentials)
  ▼
Azure AD → returns ID token + access token
  │         claims include: sub, preferred_username, upn, email, name
  ▼
BSE.Host → ASP.NET Core OIDC middleware validates token
  │         → calls UserManagement.IUserContext.ResolveGroupAsync(upn)
  │           → calls GetGroupForUser SP (existing SP, unchanged)
  │           → returns UserGroup integer + group name string
  ▼
Claims principal enriched with BSE group role claim
  → stored in session cookie (encrypted, server-generated)
```

### Identity Continuity Strategy

The `User` table currently stores `NTLogin VARCHAR(25)` in `DOMAIN\username` format. OIDC tokens carry the user's UPN (`user@domain.gov.uk`).

**Transition approach (implemented in Slice 3 — User Management):**

1. A `UPN VARCHAR(100) NULL` column is added to the `User` table via a preparatory, non-breaking database migration.
2. `IUserContext.ResolveGroupAsync(upn)` resolves identity by UPN first; falls back to NTLogin substring match for records not yet migrated.
3. A one-off data migration script (`UpdateUserUpnFromNtLogin.sql`) is provided in Slice 3 to back-populate UPN values from existing NTLogin records by mapping through AD.
4. Once all `User` records have UPN populated (verified by a CI check query), the NTLogin fallback code path is removed in a Phase 4 clean-up task.

### Authorisation Policy Registration

Five named `AuthorizationPolicy` definitions replace all per-page group-name string comparisons:

| Policy | Allowed groups |
|--------|---------------|
| `BseViewer` | All five groups (authenticated = read access) |
| `BseDataEntry` | DEFRA Data Entry, VLA Data Entry, DEFRA Maintenance, VLA Maintenance |
| `BseMaintenance` | DEFRA Maintenance, VLA Maintenance |
| `BseVlaMaintenance` | VLA Maintenance only |
| `BseDefraMaintenance` | DEFRA Maintenance only |

Every Razor Page (replacing each ASPX page) carries an explicit `[Authorize(Policy = "...")]` attribute. A Roslyn analyser rule (enforced in CI) blocks merge of any Razor Page missing an `[Authorize]` attribute.

### Configuration

OIDC configuration is entirely from environment variables (no hardcoded values in source code):

| Variable | Description |
|----------|-------------|
| `OIDC__Authority` | Azure AD authority URL: `https://login.microsoftonline.com/{tenantId}/v2.0` |
| `OIDC__ClientId` | Application (client) ID registered in Azure AD |
| `OIDC__ClientSecret` | Client secret (injected via secret store — never committed to source) |
| `OIDC__CallbackPath` | Default: `/signin-oidc` |

The Azure AD application registration requires:
- Redirect URI: `https://{app-host}/signin-oidc`
- Logout URL: `https://{app-host}/signout-oidc`
- ID token claims: `upn`, `preferred_username`, `email`, `name`
- No additional API permissions beyond OpenID Connect scopes (`openid`, `profile`, `email`)

## Options Considered

| Option | Rationale for rejection |
|--------|------------------------|
| Windows Integrated Authentication (retain) | Not available in Linux container; IIS dependency; not load-balancer compatible (see Context) |
| ASP.NET Core Windows Auth (Windows host only) | Requires Windows host — incompatible with container-based Linux deployment target |
| Azure AD B2C | Designed for external/consumer users; APHA users are internal domain accounts — Entra ID (Azure AD) is the correct provider for internal government users |
| SAML 2.0 | Older protocol; OIDC is preferred for new government applications; Azure AD has superior OIDC support |
| API keys / custom auth | Not appropriate for a user-facing government application with existing Windows domain accounts |

**OpenID Connect with Azure AD (Entra ID)** is the only option compatible with all constraints: Linux container, .NET 10, existing Windows domain accounts, government security standards, and horizontal scaling.

## Consequences

### Positive

- Authentication is compatible with Linux containers and horizontal scaling.
- Standards-based OIDC token handling is handled by `Microsoft.AspNetCore.Authentication.OpenIdConnect` — no bespoke authentication code.
- The same Windows domain credentials used by the legacy system are reused via Azure AD (no credential migration or new password creation for users).
- Authorisation defects from ADR-003 are corrected: centralised `AuthorizationPolicy` replaces per-page string comparison; role claims are populated from database group at login; no session-based identity string.
- The `GetGroupForUser` stored procedure is preserved unchanged — group names and UserGroup integers remain compatible.
- An OIDC-compatible application can be audited and monitored via Azure AD sign-in logs.

### Negative and Trade-offs

- **Azure AD dependency:** The application now has a runtime dependency on Azure AD availability. An Azure AD outage prevents all user logins. This risk is accepted as Azure AD SLA is 99.99%.
- **Application registration required:** An Azure AD application registration must be created and maintained by a tenant administrator. This is a one-time action but requires coordination with the Defra/APHA IT department.
- **Identity transition complexity:** The `NTLogin` → UPN migration requires a data migration script and a transitional fallback code path. This must be carefully managed in Slice 3.
- **Cookie-based session vs NTLM:** The legacy system used connection-level NTLM tokens (invisible to users). OIDC uses a browser redirect flow. Users will see a login page on first access. Subsequent accesses use a session cookie. This is a visible UX change.
- **Group name preservation:** The five legacy group names (`DEFRA Viewer`, `DEFRA Data Entry`, `DEFRA Maintenance`, `VLA Data Entry`, `VLA Maintenance`) are preserved in the database `luUserGroup` table and in the `UserGroup` enum. No group restructuring is in scope for this migration.

### Downstream Impact on Implementation Agent

- No ASP.NET Core middleware or page handler may call `WindowsIdentity.GetCurrent()` or use `System.Security.Principal.WindowsIdentity` for user resolution.
- `IUserContext` from `BSE.Modules.UserManagement` is the only permitted source of user identity and group membership in application code.
- OIDC middleware must be registered in `BSE.Host/Program.cs` before `UseAuthorization()`.
- Every Razor Page must declare `[Authorize(Policy = "...")]`. The default page model base class should enforce this via a base `[Authorize]` attribute.

### Downstream Impact on UI Agent

- All pages inherit from a base `PageModel` that has `[Authorize]` as a minimum.
- Role-based display logic (e.g., showing/hiding admin buttons) must use `User.IsInRole("VLA Maintenance")` or `IAuthorizationService.AuthorizeAsync()` — no string comparison of session variables.
- Page-level group restrictions are expressed as `[Authorize(Policy = "BseMaintenance")]` — not as imperative redirect checks in `OnGet()` handlers.

### Impact on ADR-003

ADR-003 documented the implicit legacy decision to use Windows Auth. That decision is now **superseded** by this ADR. ADR-003 remains in the repository as a record of the as-is state and the rationale for why Windows Auth was used. Its status should be updated to `Superseded by ADR-006`.

---

## Review Date

This decision should be reviewed if:
- The hosting environment changes from Linux containers to Windows-based hosting.
- Azure AD / Entra ID is replaced or decommissioned by APHA/Defra.
- A requirement arises for external (non-domain) user access, which may require Azure AD B2C or a separate identity provider.
