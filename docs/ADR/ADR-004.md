# ADR-004: Shared Static Data Access Class with Per-Call Connection Management

**Date:** Discovered 2026-07-20 (implicit decision, original date unknown)  
**Status:** Accepted (implicit — no record of formal decision)  
**Deciders:** Original development team (PSH, ~2003)

---

## Context

The BSELib business logic classes need to call the database. A decision was needed about how connections are managed: per-call, per-request, per-class-instance, or via a shared pool.

Evidence in code:
- `legacy/DataAccessLib/clsDataAccess.vb` — `DataAccess` class header comment: *"This class has Shared (static) methods which means the class does not need to be instantiated to be used."*
- `DataAccess.m_strConnection` is a `Private Shared` field: set once at first call via `Initialise()` (SyncLock protected), then reused for the lifetime of the application.
- Every public method in `DataAccess` (`ExecuteNonQuery`, `FillDataTable`, `FillDataSet`, etc.) creates a new `SqlConnection`, opens it, executes the command, and closes the connection in a `Finally` block.
- `legacy/DataAccessLib/clsTBCultureDA.vb` — `TBCultureDA` is also a static (Shared-method) class. BSELib classes import it as `Imports libDataAccess.libDataAccess.TBCultureDA`, making the static methods callable without qualification.
- `clsBatch.CreateBatchNumberLink()` has an overload accepting an external `SqlConnection` + `SqlTransaction`, demonstrating that client-managed transactions were anticipated but are not the primary pattern.

## Decision

Database connections are not injected, pooled by application code, or shared between calls. Each data access method creates, opens, and closes its own `SqlConnection`. The connection string is cached as a static field on first call. ADO.NET connection pooling (managed by `SqlClient`) handles physical connection reuse transparently.

## Consequences

**Observed positive effects:**
- Simple, self-contained implementation. No connection lifecycle management required in calling code.
- ADO.NET connection pooling provides de-facto connection reuse.
- Thread safety for the connection string is handled by the `SyncLock` in `Initialise()`.

**Observed negative effects:**
- **No dependency injection:** `DataAccess` is a static class with no interface and no constructor. It cannot be mocked or replaced in tests without static mocking tools or code changes.
- **No unit testability:** BSELib domain classes (`clsCase`, `clsFarm`, etc.) have no seam for injecting a test data source. Any test of business logic requires a live database connection.
- **Deprecated API:** `ConfigurationSettings.AppSettings` (used in `DataAccess.Initialise()`) was deprecated in .NET 2.0 in favour of `ConfigurationManager.AppSettings`. This compiles with a warning suppressed by `<NoWarn>`.
- **No cross-call transaction scope:** The primary pattern (each method creates its own connection) means that `clsCase.UpdateCaseDetails()`, which calls multiple stored procedures sequentially (AddCase, then child SPs), does so across separate connections and separate implicit transactions. A failure after `AddCase` but before a child SP completes leaves the database in a partially updated state. The alternative overload with an external connection/transaction is used only in `clsBatch.CreateBatchNumberLink()`.
- **Static field mutation:** `m_strConnection` is writable (no `ReadOnly`) after `Initialise()`. Although the `SyncLock` prevents a race at startup, there is no mechanism to detect if `Web.config` changes at runtime.
